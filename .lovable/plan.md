

# Plan: Dynamic AI-Generated Use Case Detail Content

## Problem
The UseCaseDetail page (sections 2-9: detection signals, scope, detailed steps, exceptions, before/after, value metrics, traceability) currently uses hardcoded mock data from `useCaseDetailData.ts`. The user wants this content to be **dynamically generated by the AI** during the `analyze-process` step and stored in the database.

## Approach

### 1. New Database Table: `use_case_details`

Store all enriched detail sections as a single JSONB column per use case, following the existing `UseCaseDetail` TypeScript interface structure:

```sql
CREATE TABLE public.use_case_details (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  use_case_id uuid NOT NULL REFERENCES automation_use_cases(id) ON DELETE CASCADE,
  detail_content jsonb NOT NULL DEFAULT '{}'::jsonb,
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE(use_case_id)
);
-- RLS: public read/insert/update/delete
```

The `detail_content` JSONB will contain all sections: `detectionSignals`, `contextReferences`, `confidenceLevel`, `confidenceExplanation`, `willBeAutomated`, `willRemainManual`, `explicitExclusions`, `detailedSteps`, `exceptions`, `comparison`, `valueMetrics`, `traceabilityLinks`.

### 2. Update Edge Function: `analyze-process`

Add a **second AI call** after use cases and variants are generated. For each use case, call the AI with the process context + use case info and ask it to generate the full detail content matching the template structure (all 7 enriched sections). Store the result in `use_case_details`.

The AI tool schema will mirror the `UseCaseDetail` interface exactly — detection signals with type/label/description/triggeringStep/ruleOrThreshold, scope items, detailed automated steps, exception scenarios, before/after comparison, value metrics, and traceability links.

### 3. Update `UseCaseDetail.tsx` Page

- Add a query to fetch `use_case_details` for the current use case ID
- Use DB detail data when available, fall back to mock data only for mock IDs (uc1-uc7)
- No UI template changes needed — the existing rendering code stays identical, just the data source changes from `mockUseCaseDetails[id]` to the DB record's `detail_content`

### Files to Create
- **Migration**: New `use_case_details` table with RLS policies

### Files to Edit
- `supabase/functions/analyze-process/index.ts` — add second AI call to generate detail content per use case, store in `use_case_details`
- `src/pages/UseCaseDetail.tsx` — fetch `detail_content` from DB, fall back to mock only for mock IDs

### Technical Notes
- Using a single JSONB column keeps the schema simple and matches the complex nested structure (arrays of objects with varied fields)
- The AI will generate content in French to match the existing UI language
- The `analyze-process` function will process details sequentially per use case to avoid rate limits
- Mock data remains as fallback for demo/preview purposes when no DB data exists
